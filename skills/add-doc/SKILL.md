# add-doc — Source Document Management

Adds documents to the project source library with structured metadata, importance
tagging, and automatic indexing. Manages the `sources/` directory tree so source
documents are organized, discoverable, and immediately available to the search
and build pipeline.

---

## When to Use

- When the user provides a new source document (PDF, PPTX, DOCX, HTML, image,
  spreadsheet) to inform the deck.
- When source documents arrive mid-session and need to be incorporated.
- When the user says "add this document" or "include this file".
- During Phase 1 (Source Inventory) of the workflow to build the source library.

---

## Supported File Types

| Type | Extensions | Routing Subfolder |
|------|-----------|-------------------|
| **Reports & analysis** | `.pdf` | `sources/reports/` |
| **Existing presentations** | `.pptx`, `.ppt` | `sources/decks/` |
| **Written documents** | `.docx`, `.doc`, `.rtf` | `sources/documents/` |
| **Web content** | `.html`, `.htm` | `sources/web/` |
| **Images & diagrams** | `.png`, `.jpg`, `.jpeg`, `.svg`, `.gif` | `sources/images/` |
| **Spreadsheets & data** | `.xlsx`, `.xls`, `.csv` | `sources/data/` |
| **Other** | Any unrecognized extension | `sources/other/` |

Routing is automatic based on file extension. If the user specifies a different
subfolder, respect their choice.

---

## Importance Levels

Every document gets an importance tag that affects how it is used downstream:

| Level | Meaning | Indexing Behavior | Examples |
|-------|---------|-------------------|----------|
| **critical** | Must-read for every agent, core to the deck's purpose | Full indexing, included in facts manifest summary | RFP document, board mandate, primary strategy doc |
| **high** | Key source, frequently referenced | Full indexing, available to all search scopes | Financial reports, competitive analysis, prior decks |
| **standard** | Useful reference material | Full indexing, normal search ranking | Industry reports, background reading, org charts |
| **reference** | Supplementary, consult if needed | Full indexing, lower search ranking | Style guides, glossaries, templates, general context |

Importance is assigned based on user input. If the user does not specify, infer
from context:
- Documents provided first or described as "the main" source → **critical**
- Documents the user explicitly calls important → **high**
- Documents provided without emphasis → **standard**
- Documents described as "for reference" or "background" → **reference**

When uncertain, ask. Do not default to critical.

## Metadata Sidecar

For every added document, write a sidecar file at `{document_name}.meta.md`
in the same directory as the document. Format:

```markdown
---
importance: high
added: 2026-03-23T14:30:00Z
source_file: quarterly-review-q3.pdf
description: Q3 financial review with revenue breakdown and forecast updates
tags: [financial, quarterly, forecast]
added_by: session
---

# quarterly-review-q3.pdf

Q3 financial review with revenue breakdown and forecast updates.

## Key Content
- Revenue figures by business unit
- Year-over-year comparisons
- Updated forecast for Q4 and FY

## Notes
{Any user-provided context about why this document matters}
```

The `Key Content` section is a brief summary (3-5 bullets) written after
scanning the document's first few pages. This provides a quick reference
without opening the full document.

## Workflow

```
User provides document path
    │
    ▼
Validate file exists and type is supported
    │
    ▼
Determine routing subfolder (by extension)
    │
    ▼
Create subfolder if it does not exist
    │
    ▼
Copy or move file to sources/{subfolder}/
    │
    ▼
Assign importance (user-specified or inferred)
    │
    ▼
Scan first pages → write sidecar .meta.md
    │
    ▼
Check gaps.md for items this document may resolve
    │  ├─ match found → report to user, mark as potentially resolved
    │  └─ no match → continue
    │
    ▼
Auto-invoke wicked-prezzie:learn
    │  ├─ Pass 1: index the new document only (incremental)
    │  └─ Pass 2: full rebuild of cross-document synthesis
    │
    ▼
Report: document added, location, importance, index status, any gaps resolved
```

---

## Gap Resolution

If the project has a `gaps.md` file (generated by learn Pass 2 at
`index/_insights/gaps.md`), check whether the new document addresses any
listed gaps.

**Matching procedure**:
1. Read `gaps.md` for the list of coverage gaps
2. Scan the new document's content and sidecar tags
3. If a gap topic appears substantively in the new document, report it:
   "This document may resolve the gap: '{gap description}'. Verify after
   indexing completes."
4. Do not auto-remove gaps — learn Pass 2 will update `gaps.md` after
   re-synthesis

## Invocation Examples

```
"Add this PDF to the project"
  → add-doc(path="~/Downloads/market-analysis.pdf")
  → routes to sources/reports/, importance: standard, auto-indexes

"This is the main strategy document, critical"
  → add-doc(path="strategy-2026.docx", importance="critical")
  → routes to sources/documents/, importance: critical, auto-indexes

"Include these slides for reference"
  → add-doc(path="prior-deck.pptx", importance="reference")
  → routes to sources/decks/, importance: reference, auto-indexes

"Add this spreadsheet with the financial data"
  → add-doc(path="financials.xlsx")
  → routes to sources/data/, importance: standard (inferred), auto-indexes

"Import these three documents"
  → add-doc called once per document, each routed and indexed
```

---

## Batch Mode

When the user provides multiple documents at once:

1. Process each document sequentially (validate, route, write sidecar)
2. Run learn Pass 1 for each new document (can be parallelized)
3. Run learn Pass 2 once after all documents are indexed (single rebuild)
4. Report summary: documents added, locations, importance levels, any gaps resolved

This avoids running Pass 2 multiple times for a batch import.

## Limitations

- File type detection is extension-based only. A `.pdf` that is actually a
  renamed `.docx` will be routed to `sources/reports/` and may fail during
  learn indexing. The learn skill will report the error.
- Sidecar generation scans the first few pages only. For large documents,
  the `Key Content` summary may miss material from later sections. The full
  content is captured during learn indexing.
- Duplicate detection is name-based. If the same document is added twice with
  different filenames, both copies will be indexed. Check the index manifest
  for duplicate chunk content after batch imports.
